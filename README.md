## 教育支援キャラクター育成アプリ（ハッカソン）

> **マージしたあと、みんなで同じ環境で動かす手順**は **[マージ後の環境構築手順.md](./マージ後の環境構築手順.md)** にまとめてあります。プログラミングに慣れていない人向けに、番号どおりにやればよいように書いてあるので、まずはそちらを開いてください。

### プロジェクト概要

- **目的**: 小〜高校生向けの学習・生活記録アプリ。  
  - ユーザーが「学習状況」「趣味」「一日の生活」「学年」などを入力すると、それに応じた**オリジナルキャラクター**が育成される。  
  - キャラクター画像は **Gemini API の画像生成**を使い、ユーザー情報に関連した見た目にする。  
    - 例: 「サッカーが好き」「化学が得意」な男の子 → 白衣を着てサッカーボールを持っている少年キャラ。

- **対象**: 小学生〜高校生  
  - 画面に表示するテキストの漢字レベルを「学年ごと」に調節できるようにする（低学年はひらがな多め、高学年は漢字多めなど）。

- **主要画面（フッターナビ）**
  - **ホーム**: キャラクター表示・ステータス確認
  - **記録**: 学習状況・趣味・生活リズムなどの入力フォーム
  - **ニュース**: ユーザーの興味・学習分野に関連するニュース一覧
  - **設定**: ユーザー基本情報・学年・漢字レベル設定など

---

### 使用技術

- **バックエンド**
  - PHP（シンプルな自作ルーター + Firebase 連携）
  - ※時間に余裕があれば Laravel 化も検討

- **フロントエンド**
  - HTML / CSS
  - JavaScript (jQuery)

- **データベース**
  - Firebase Realtime Database（`kreait/firebase-php` を利用）

- **外部API**
  - Gemini API（画像生成）  
    - 本番前まではダミー画像で実装し、最後に差し替える運用でもOK。

---

### ディレクトリ構成（抜粋）

- `public/`
  - `index.html` … フロントエンドのメイン画面（フッターナビ付き・シングルページ想定）
  - `backend/` … PHP バックエンド（Firebase 連携 API）
    - `index.php` … シンプルなルーター（`/api/...` を振り分け）
    - `src/Service/MemoService.php` … メモ保存用の Firebase ラッパ
    - `src/Service/CharacterService.php` … キャラクター関連のデータ取得・保存（予定）
    - `src/Controller/MemoController.php` … メモ API のコントローラ
    - `src/Controller/CharacterController.php` … キャラクター API のコントローラ（予定）

### 4人チームの役割分担（例）

#### バックエンド1（BE1）: API & データ設計担当

- **担当範囲**
  - PHP バックエンドのルーティングと API 設計
  - 「キャラクター」「記録」「設定」などの**データ構造の定義**
  - Firebase との連携部分（サービスクラス）の実装
- **具体的タスク**
  - `/api/character`, `/api/records`, `/api/settings` などのエンドポイントを設計・実装
  - ユーザーIDごとのデータ保存ルールを決める（例: `users/{userId}/records`, `users/{userId}/character` など）
  - テスト用ダミーデータ・モックレスポンスの作成

#### バックエンド2（BE2）: Gemini & ビジネスロジック担当

- **担当範囲**
  - Gemini API 呼び出し部分の実装
  - 入力情報 → キャラクター説明文（プロンプト）への変換ロジック
  - キャラ画像生成のトリガー・再生成ロジック
- **具体的タスク**
  - 「学習状況」「趣味」「生活リズム」「学年」などを組み合わせて**プロンプト文章**を生成
  - Gemini のレスポンス（画像URLなど）を Firebase に保存する処理
  - 失敗時のリトライやエラーメッセージの設計

#### フロントエンド1（FE1）: 画面デザイン & UI/UX 担当

- **担当範囲**
  - 画面のレイアウトとスタイル（HTML/CSS）
  - フッターナビゲーション（ホーム / 記録 / ニュース / 設定）
  - 学年ごとの漢字レベルに応じたテキストや表現の検討
- **具体的タスク**
  - 各セクション（ホーム・記録・ニュース・設定）のコンポーネントをデザイン
  - レスポンシブ対応（スマホ優先）
  - カラー・フォント・ボタン・カードレイアウトの共通デザイン決め

#### フロントエンド2（FE2）: フロントロジック & API 連携担当

- **担当範囲**
  - jQuery によるフォーム送信・バリデーション
  - バックエンド API との通信（Ajax）
  - 取得したキャラ画像やニュースの画面反映
- **具体的タスク**
  - 「記録」フォームから `/api/records` に POST する処理
  - 「ホーム」画面で `/api/character` からキャラ情報を取得・表示
  - 「設定」変更時に `/api/settings` を叩いて保存
  - ローディング表示・エラー表示などのユーザー体験向上

### 画面ごとの担当とやること

- **ホーム画面（キャラクター）**
  - **BE**: `/api/character` でキャラ情報 + 画像URL + ステータスを返す。
  - **FE**: キャラ画像・名前・ステータスバーなどを表示。キャラ再生成ボタンなど。

- **記録画面**
  - **BE**: `/api/records` で学習・生活の記録を保存。必要に応じてキャラ更新フラグを立てる。
  - **FE**: 学習時間・教科別の出来具合・趣味・睡眠時間などのフォーム UI 実装。

- **ニュース画面**
  - **BE**: ユーザーの興味タグ（例: サッカー/化学）に応じたニュース一覧を返す。初期はモック配列でも可。
  - **FE**: ニュースカード表示・リンク遷移の実装。

- **設定画面**
  - **BE**: ユーザー基本情報（学年・ニックネーム・漢字レベル設定）の保存/取得。
  - **FE**: プロフィール編集フォームと保存ボタンの実装。

### 開発フロー（例）

1. **Day 1**
   - 全員で画面遷移図・ワイヤーフレームを決める。
   - BE1/BE2: API 一覧とデータ構造の草案を作成。
   - FE1/FE2: 画面レイアウトのたたきを作る。

2. **Day 2**
   - BE1: `/api/character`, `/api/records`, `/api/settings` の**ダミー実装**（固定データを返す）を作る。
   - BE2: Gemini API の試験実装（固定プロンプトで画像生成ができる状態まで）。
   - FE1/FE2: ダミー API とつないで、画面が一通り動く状態まで持っていく。

3. **Day 3 以降**
   - BE 側: 実データ + Firebase 連携に差し替え。
   - FE 側: 漢字レベル・学年切り替え UI、デザイン調整、バグ修正。
   - 最後に: プレゼン資料・デモフローの確認。

### Git / ブランチ運用（シンプルルール）

- **main ブランチ**
  - 常に動く状態。発表用ブランチ。

- **feature ブランチ**
  - 機能ごとに分ける。例:
    - `feature/backend-character`
    - `feature/backend-records`
    - `feature/frontend-home`
    - `feature/frontend-settings`

- **プルリクエスト**
  - 小さめの変更単位でプルリクを作成。
  - 少なくとも 1 人にレビューしてもらう。
  - 同じファイルを同時に編集しすぎないように注意する。
